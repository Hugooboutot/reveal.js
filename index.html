La FR adoooore. * **Java EE (JSP, Servlets, EJB, Hibernate…) :** le royaume de la sur-architecture et des process qui étouffent. « Agile » (se méfier du A majuscule) l’oxymore. Par ailleurs, l’écosystème ASP.NET est souvent à la traîne en termes de paradigmes… Et les machines Windows sont juste ultra-pénibles pour tout autre outil de dev que VS. * **Rails** : La killer app pour Ruby. A remis sur le devant de la scène énormément de meilleures pratiques (CoC, MVC, ActiveRecord, REST, TDD/BDD, Asset Pipeline, Performances…). Utilisé par la très grande majorité des startups et boîtes Web 2.0 de 2006 à 2011, encore extrêmement populaire. * **Django** : Similaire à Rails, mais basé sur Python, et moins avancé/réactif au début. Très populaire. La communauté Python est une des plus sympas, ouvertes, etc. qui soit. * **Node** : La grosse révolution de ces 5 dernières années. L'outil de choix pour la plupart des dernières startups. Très bien adapté au temps réel et très fortes charges. A ouvert la boie du JS isomorphique et de l'outillage dev basé JS. Est en train d'éjecter Java EE de nombreuses très grosses boîtes renommées.

Aplanir

Les différences

JS et DOM : le sol n’arrête pas de bouger :-(

2005–2009 : Prototype.js

Depuis : jQuery

jQuery ≠ JavaScript !@#

Quant à YUI, Dojo, Mootools, Ext… OSEF.

On ne parle plus tellement de Prototype, jQuery lui a pris la place, mais Prototype avait tout inventé :-) Il est impératif de bien distinguer jQuery, qui est une lib pour aplanir les soucis cross-browser (surtout desktop) et faciliter Ajax + DOM, de JavaScript. jQuery cherche à isoler de JS autant que possible, et la grande majorité des jQueristes (utilisateurs de jQuery) sont des brèles en JS. Surtout, surtout, sur vos CV, évitez l'atroce « JavaScript (jQuery) ». C'est un peu comme dire « Shakespeare (LOL) ». Il y a eu pas mal d'autres libs (Mootools, ces p'tits salopiaux) et quelques frameworks (YUI, Dojo, Ext), dont certains perdurent encore. Mais ils sont remplacés au fil de l'eau par des trucs plus modernes et sont ressortis du radar.

Le web mobile

Plus de 50% du trafic web début 2015.

Toutes les estimations voient ça finira par être ~90%.

Les browsers mobiles sont excellents*
beaucoup WebKit, plus Fennec (Firefox Mobile) et IE Mobile

* sauf le browser par défaut d’Android pre-Lollypop (« IE6 du mobile »)

Énorme marché des apps mobiles hybrides (99% web dans une coquille native vide), au travers notamment de PhoneGap et Titanium.

Le futur est illustré par des projets comme FirefoxOS.

Côté technos, le web mobile est nettement plus à jour que le web desktop. Chaque plate-forme a au moins un browser de tueurs à disposition, bien répandu. On est plus limités par la puissance pure des smartphones et tablettes, leurs capacités RAM et GPU, leurs plafonds de cache, etc. Les capacités « natives » (notifications, accès aux contacts, à la webcam, au micro, au vibreur…) ne deviennent que lentement accessibles en pur web (Device APIs) ; en attendant, on peut jouer la carte hybride pour rester massivement basé web et contourner ainsi les galères de validation sur les Stores.

Libs incontournables

Pour ne pas réinventer la roue

Underscore pour l’algo courante (ou Lo-Dash)

Moment (pour toute manip date/heure)

Préprocesseurs CSS : LESS, SASS, Stylus…

Frameworks CSS : Bootstrap, Compass, SemanticUI…

Underscore est partie du corpus de méthodes algos de Prototype.js et l'a rapidement étendu, pour ajouter de la programmation fonctionnelle notamment. C'est le complément indispensable à jQuery, qui a permis aux Prototypistes de migrer, notamment. Dépendance n°1 dans l'univers JS. Lo-Dash est parfois jugée plus « hype » mais n'est même pas plus rapide aujourd'hui (même si elle a parfois une API plus sympa). Je préfère rester sur Underscore, personnellement. Les dates en JS sont un clusterfuck (voir article JS Attitude), et Moment est au top. Formate, parse, calcule des deltas fins, et tout type de manip date/heure imaginable, de façon fiable et internationalisée. Au top ! Faire des CSS brutes, « à mains nues », n'a plus aucun intérêt. Un préprocesseur est devenu indispensable pour gagner en productivité et en maintenabilité. LESS à négliger, préférez SASS voire Stylus. Il est également préférable de ne pas réinventer la roue côté layout et responsive, en s'appuyant sur un framework établi, propre, maintenu et bien répandu / battle-tested. Bootstrap est extrêmement populaire, mais j'aime bien ce que fait SemanticUI ; on a aussi des trucs dédiés, genre Topcoat pour le mobile pur en perfs poussées.

Webperf

Pour une poignée de millisecondes de moins

Vos pages doivent se charger très, très vite
(et pas que sur votre Mac via fibre : sur smartphone via 3G/Edge)

L’immense majorité des sites sont mauvais en perfs.
Il faut savoir jouer avec plein de variables et d’optimisations : concaténation / spriting, compactage / subsetting, GZipping, cache & CDN, DNS prefetching, trimming CSS, optis JS et DOM…

Impact financier fort : outre les coûts de stockage / BP, le taux de conversion en e-commerce est fortement corrélé. On peut perdre des millions par an juste parce qu'on met 100ms de trop en moyenne à charger…

* Amazon a fait +1% de CA en gagnant 100ms de chargement. * Bing a perdu 250K$/an en mettant 10ms de plus à charger. * Charger en 1s au lieu de 200ms fait -3,5% de CVR (Conversion Rate) et -2,1% de PM (Panier Moyen). * Frontière du TTG (Time To Glass) : ~1s en desktop, ~4s en mobile.

Principes

fondamentaux du dev web

Page web = HTML + CSS + JS (et DOM)

Web = plate-forme universelle. JS = langage universel*

La frontière page / app / service se brouille, disparaît. « Handovers » d'un périphérique à l'autre.

Déploiement fluidifié par rapport au desktop.

Analyse temps réel des utilisateurs et de leurs comportements : opportunités d'itération rapide.

* et non Java ou C#
Derrière JS et DOM se cachent, on l'a vu, en fait énormément d'APIs et de possibilités. Une page, ça va d'un formulaire de contact à Assassin's Creed: Pirates, en passant par Gmail, Facebook et Cloud9 ! JS est mono-thread, non-réentrant (mais workers) : pas de soucis de multi-threads, sémaphores et mutex, pas de perte de temps à attendre une I/O assis sur ses mains… Au cœur de la performance de Node.js. Web vs. desktop : pas d'installation, pas d'envoi de mises à jour, donc pas de piratage. Modèles de type SaaS, par abonnement, revenus passifs / mécaniques souvent à croissance stable. Pléthore de services d'analyse, temps réel ou non, et d'interaction avec les utilisateurs. A/B testing. Déploiement continu et feature flags. Montée à l'échelle horizontale obtensible en respectant soigneusement une architecture de type 12 Factors.

L’âge de raison

2005–2010

À mort XML, SOAP et WS-*, vive JSON et REST !

Ajax et complexification / enrichissement de la couche client

Google Maps, Gmail, Netvibes, Twitter, Facebook… De vraies applis dans le browser.

La peste née en 1998 qui voulait le tout-XML, WSDL, ebXML, SOAP, etc. et a fait crouler tant de système sous leur propre poids commence à avoir un sérieux coup de plomb dans l'aile. Dès 2006, les gros acteurs (Amazon, eBay, Yahoo!…) lâchent ces saletés pour faire du REST du JSON, infiniment plus pratiques et souples. Ajax décolle, mais sans le X : on fait transiter du HTML, du JSON et du JS. Le monde commence à réaliser qu’il est possible de faire des trucs de dingue, avec des codebases considérables, en JS pur.

Le browser comme plate-forme

Pourquoi du MVC côté client ? On revient au client riche, qui permet plus de réactivité, l'offline, le côté vraiment applicatif. On développe des apps complètes dans le browser, le serveur devient un pur backend, une API. Évitez Angular. Sérieusement, FBI. L'écosystème de tests est ultra-riche et moderne, et les tests sont devenus une vraie religion dans le dev front, et apportent les avantages énormes habituels. ES5 bien présent sur tous les browsers modernes (tous + IE9+), et ça aide. JS isomorphique : la possibilité croissante de coder un seul JS qui marche aussi bien côté serveur (Node.js notamment) que côté client, donc partage de codebase, dette technique réduite, meilleure capitalisation des connaissances. JS runtime : optis via asm.js, LLJS, Emscripten ; démo Citadel, UE4 et AC:Pirates 100% JS/WebGL…

Le backend est de plus en plus relégué au rang de serveur API et service de stockage, qu'on peut donc génériser, toute la valeur métier résidant dans l'appli côté client. JavaScript vs. Java

Un nommage malheureux (alors qu’on fait pas plus différents)

Java JavaScript
Typage Statique Dynamique
Disponibilité Presque partout, à installer Partout, de base
Développement Surtout propriétaire, lent depuis 2002 Standard et libre, rapide depuis 2009
Exécution Lourde, voire très lourde Légère, voire très légère
Utilisation Back-end (applets mortes) Full stack

Mais au-delà, rien. Et pendant ce temps-là, vous n'aurez pas pu acquérir les savoirs qui vous seront nécessaires pour pivoter vers le marché qui aura tout dominé : le web-based, dont JS est la lingua franca…

Fonctions de 1er ordre


function foo() { … }
var f = foo;

var obj = { yo: foo };
            

var obj1 = {
  run: function run() { … }
};

var obj2 = {
  run: obj1.run
};
Une fonction est une valeur comme les autres : elle peut être référencée sans être appelée (sans les parenthèses, donc). Dans d'autres langages, on parlerait de pointeur sur fonction. En Java, ce qui s'approche le plus de ça, ce sont les lambdas de Java 8, mais sinon on n'a rien. Les fonctions sont toutes « libres » : elles n'appartiennent à aucun objet en particulier. Le fait qu'un objet ait une propriété avec la fonction comme valeur *ressemble* à une méthode, et se comporte comme tel, mais c'est du confort : la fonction n'est pas intrinsèquement liée à l'objet.

      return result;
    }
    result = fx.apply(this, arguments);
    called = true;
    return result;
  };
}
            

var index = 0;
function demo() { return ++index; }
var memoDemo = memoize(demo);
memoDemo(); // => 1
memoDemo(); // => 1
memoDemo(); // => 1

function makeSpy(fx) {
  var spy = function spy() {
    var call = {
      context: this, args: arguments, stamp: Date.now(),
      result: fx.apply(this, arguments)
    };
    spy.calls.push(call);
    return call.result;
  };
  spy.calls = [];
  spy.displayName = "spy" + (fx.name || "");
  return spy;
}
            

var obj = {
  greet: function greet(whom) { console.log('Hi', whom); return 42; }
};
obj.greet = makeSpy(obj.greet);
obj.greet('John'); // => 42
obj.greet.calls
// => [{ context: (obj), args: ['John'], stamp: 14219478…, result: 42 }];

Quelques exemples


function createDefensiveObject(target) {
  return new Proxy(target, {
    get(target, property) {
      if (property in target) {
        return target[property];
      }

      throw new ReferenceError(`Property "${property}" does not exist.`);
    }
  });
}
            

function Person(first, last) {
  this.first = first;
  this.last = last;
  return createDefensiveObject(this);
}

var protector = new Person('Serge', 'Karamazoff');
protector.age // => ReferenceError: Property "age" does not exist.
